# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

This project is an MCP server that wraps [Hayabusa](https://github.com/Yamato-Security/hayabusa) for EVTX (Windows Event Log) analysis. It exposes two MCP tools:

- `scan_evtx` — runs Hayabusa's `json-timeline` command against an EVTX file and returns findings as structured JSON, with filtering by severity and rule title.
- `get_hayabusa_rules` — lists/searches the bundled Sigma-style detection rules (title, ATT&CK tags, severity, description) without running a scan.
- `analyze_coverage` — takes an ATT&CK technique ID (`T1003`, `T1003.001`) or tactic name/slug (`credential-access`, `Lateral Movement`) and returns a coverage report (per-technique `covered`/`partial`/`gap` plus summary counts) against our **custom** rules, reusing the same assessment logic as the `detection://attack/techniques/{id}` resource.
- `suggest_rule` — takes a leaf ATT&CK technique ID, checks our custom-rule coverage for it, and if it isn't already `covered`, suggests a detection approach (grounded in how the vendored builtin rules already detect it, or a rough keyword-based logsource guess if none do) and can optionally write a draft Sigma rule template into `rules/`.

It also exposes MCP resources over the project's own custom rules in `rules/` (not the vendored builtin set):

- `detection://rules` — list all custom rules (metadata only).
- `detection://rules/{rule_name}` — raw YAML content of one rule, keyed by filename stem (e.g. `security_dcsync_replication_rights`).
- `detection://rules/by-technique/{technique_id}` — rules tagged with a given ATT&CK technique; accepts a sub-technique (`T1003.001`) for an exact match or a parent technique (`T1003`) to match all its sub-techniques.
- `detection://attack/techniques/{technique_id}` — an ATT&CK technique's name/description plus a coverage assessment (`covered`/`partial`/`gap`) against our custom rules, using a local MITRE ATT&CK technique index (see below).
- `detection://playbooks` — list all incident response playbooks in `playbooks/` (metadata only).
- `detection://playbooks/{playbook_name}` — full content of one playbook by filename stem (e.g. `credential-theft`), with its `related_rules` enriched from the live custom rule files.
- `detection://playbooks/by-alert/{alert_name}` — playbook(s) relevant to an alert/rule name, matched case-insensitively against each playbook's alert aliases and related rules' names/titles; returns an empty match list rather than an error if nothing matches.

## Commands

- Run the server (used by MCP clients via `.mcp.json`): `python3 server.py`
- Manual smoke test against the bundled sample EVTX: `python3 test_server.py`
- Install/upgrade the Hayabusa binary + rules into `./hayabusa/`: `scripts/download_hayabusa.sh` (detects OS/arch, pulls the latest GitHub release, safe to re-run)
- Fetch/refresh the local MITRE ATT&CK technique index into `./attack/`: `python3 scripts/download_attack_data.py` (downloads the ~50MB Enterprise STIX bundle, extracts a ~1MB technique index including tactics, safe to re-run)
- Install Python deps: `pip3 install -r requirements.txt`

There is no formal test suite (no pytest) — `test_server.py` is a standalone script that calls `scan_evtx` directly and prints results for manual inspection.

## Architecture

Everything lives in `server.py`, a single-file FastMCP server. Key structure:

- **`HAYABUSA_DIR` / `HAYABUSA_BIN` / `BUILTIN_RULES_DIR`**: paths into the vendored `hayabusa/` directory, which contains the downloaded Hayabusa binary (`hayabusa/hayabusa`, symlinked to the versioned executable), its `rules/` (Sigma-style YAML detection rules) and `rules/config/` (field mappings, channel abbreviations, etc. needed by Hayabusa at scan time). This directory is populated by `scripts/download_hayabusa.sh` and is not meant to be hand-edited.
- **`rules/` (repo root, `CUSTOM_RULES_DIR`)**: hand-written project-specific Sigma rules, additive to the vendored set. `RULE_SOURCES` lists both roots (`builtin`, `custom`) and is the single place both tools read from.
- **`scan_evtx` tool**: shells out to the Hayabusa binary (`json-timeline` subcommand) with a fixed set of flags (JSONL output, no wizard/color/banner/summary, clobber output file), writes results to a temp file, then parses each JSONL line back into Python. Post-processing (rule-title filtering, `max_results` truncation, dropping bulky fields for `output_format="summary"`) happens in Python after Hayabusa runs, not via Hayabusa CLI flags — `min_severity` is the one filter passed through to Hayabusa itself (`-m`). Since Hayabusa's `-r` flag only accepts a single directory and its rule loader doesn't follow symlinked subdirectories while walking it, `_merge_rule_sources` hardlinks both `RULE_SOURCES` into a fresh temp directory before each scan (hardlinks are effectively free — no file data is copied) so a single `-r` covers builtin + custom rules together.
- **`get_hayabusa_rules` tool**: reads rule YAML files directly from both `RULE_SOURCES` roots (not via the Hayabusa binary, via `_iter_rule_files`) using `yaml.CSafeLoader` when available for speed, since there are several thousand rule files. Skips `rules/config/` under the builtin root (data-mapping files, not detection rules) and any path containing `.git`. Each rule's `rule_path` in the response is relative to the repo root, so builtin and custom rules are distinguishable (e.g. `hayabusa/rules/sigma/...` vs `rules/...`).
- **Severity handling**: `SEVERITY_LEVELS` / `SEVERITY_ALIASES` normalize both full names (`informational`, `critical`) and Hayabusa's abbreviated forms (`info`, `crit`, `med`) to a canonical value before passing to the CLI.
- All user-facing errors (missing file, missing binary, invalid severity, scan timeout/failure) are raised as `mcp.server.fastmcp.exceptions.ToolError` so the MCP client gets a clean error message rather than a stack trace.
- **`detection://` resources**: scoped to `CUSTOM_RULES_DIR` only (`_iter_custom_rule_files`), separate from the tools' merged `RULE_SOURCES` view. `_parse_rule_metadata` is shared with `get_hayabusa_rules` so both surfaces describe a rule the same way. A rule's `rule_name` (used in the URI) is its path relative to `rules/` with the extension stripped, via `_custom_rule_name`. Errors (unknown rule name) are raised as plain exceptions — FastMCP's resource dispatch wraps any exception from a resource function into a client-facing error itself, so there's no resource-specific error type to use (unlike `ToolError` for tools).
- **`detection://attack/techniques/{technique_id}` resource**: reads `attack/enterprise-attack-techniques.json`, a compact technique index built offline by `scripts/download_attack_data.py` from MITRE's STIX bundle — the server never fetches or parses the full ~50MB bundle itself, and `_load_attack_techniques` caches the parsed index in a module-level global for the life of the process. Each entry also carries a `tactics` list (the STIX `kill_chain_phases` phase names, e.g. `credential-access`), used for tactic-name lookups. Coverage assessment logic lives in `_assess_technique_coverage` (shared with the `analyze_coverage` tool): for a technique with sub-techniques (e.g. `T1003`), coverage is `covered` only if every sub-technique has a matching rule, `gap` if none do, else `partial`; for a leaf technique, coverage is `covered` only if a matching rule's `status` is `stable` (a `test`-status-only match is `partial`, since none of our current custom rules have been promoted past `test`) — `gap` if there's no matching rule at all.
- **`analyze_coverage` tool**: accepts either a technique ID or a tactic name in one `identifier` argument, disambiguated by checking whether it normalizes to a known technique ID first (via `_normalize_technique_id`) and falling back to a tactic-slug match (via `_normalize_tactic_slug` against the set of tactics found in the index) otherwise. Technique-ID queries reuse `_assess_technique_coverage` directly; tactic queries iterate every technique tagged with that tactic, skipping sub-techniques whose parent is tagged with the same tactic (so they're represented once, via the parent's roll-up, not double-counted) and calling `_assess_technique_coverage` per remaining entry. Both paths funnel through `_coverage_report`, which buckets results into `covered_techniques`/`partial_techniques`/`gap_techniques` plus summary counts. Note: MITRE's live ATT&CK data no longer has a `defense-evasion` tactic — it was split into `stealth` and `defense-impairment` — while rules in this repo (and the broader Sigma corpus) still tag techniques with the old `attack.defense-evasion` label; `LEGACY_TACTIC_HINTS` surfaces a pointer to the new names when someone queries the old one. This doesn't affect technique-ID-based rule matching (which keys off stable `T\d+` IDs, unaffected by the rename), only tactic-name lookups.
- **`suggest_rule` tool**: also builds on `_assess_technique_coverage`. For a parent technique (has sub-techniques), it never suggests a rule or writes a file — it only reports which sub-techniques are uncovered and asks for a specific one, since a rule tagged with just the broad parent ID isn't a meaningful detection target. For a leaf technique that isn't `covered`, `_find_builtin_rules_by_technique` (same tag-matching as the custom-only `_find_custom_rules_by_technique`, but walking `BUILTIN_RULES_DIR` instead) looks for existing builtin rules to ground the suggestion in — the majority logsource among matches becomes `suggested_logsource`, and up to 5 reference rules (stable/critical first) are returned by path. If there's no builtin match either — a real gap in the ~5000-rule vendored corpus, not just this project — `_keyword_logsource_guess` falls back to a small keyword→logsource table matched against the technique's ATT&CK description; the response always states this is a rough guess, not a researched recommendation. `_build_rule_template` produces a draft Sigma rule dict (status `experimental` — one step below the `test` status hand-written custom rules start at, since it's unreviewed and has placeholder detection logic) tagged with the technique's tactics and ID; `create_rule_file=True` writes it via `_rule_template_filename`, which picks a filename prefix from `_LOGSOURCE_FILE_PREFIXES` (falling back to the raw category/service) plus a slug of the technique name, incrementing a numeric suffix on collision. File creation is always skipped (with a note in the response) for an already-`covered` technique or a parent-technique query, regardless of `create_rule_file`.

- **`playbooks/` (repo root, `PLAYBOOKS_DIR`)**: hand-written incident response playbooks, one YAML file per playbook, filename stem = `playbook_name` used in the resource URI (via `_playbook_name`, same relative-path-minus-extension convention as `_custom_rule_name`). Each file's `related_rules` field is a list of custom rule names (not embedded rule detail) — `_parse_playbook` resolves each against the live custom rule files via `_lookup_custom_rule` at read time, so a rule's title/level/status/tags in a playbook response can never drift from the actual rule (a rule name that no longer resolves is flagged `"rule_missing": true` instead of failing the whole playbook). `detection://playbooks` (list) strips this down to summary fields, same list-vs-detail split as the `detection://rules` / `detection://rules/{rule_name}` pair, except the detail view here returns enriched JSON rather than the raw file (playbooks are analyst-facing narrative content, not something Hayabusa itself consumes, so there's no raw-file fidelity requirement the way there is for Sigma rules). `detection://playbooks/by-alert/{alert_name}` (`_playbook_matches_alert`) does a case-insensitive bidirectional-substring match against `alert_aliases` plus an exact/substring match against related rules' `rule_name`/`title` — a "search," not a direct key lookup, so zero matches returns an empty list rather than raising, consistent with `get_hayabusa_rules(keyword=...)`'s behavior. Note when hand-writing a playbook YAML: any list item containing `": "` (a colon followed by a space) must be quoted, or YAML parses it as a single-key mapping instead of a plain string.

`credential_access_rules.json` at the repo root is a saved example output of `get_hayabusa_rules` (keyword `attack.credential-access`), not source code.
